Задача 3.2: Алгоритм обработки запроса на создание пользователя (бэкенд)

Документ описывает внутреннюю логику работы бэк-сервиса при получении HTTP-запроса `POST /api/v1/auth/register` на основе предоставленных в ТЗ UI-прототипов.

## 1. Входной контроль и Валидация данных
При получении JSON-пакета сервер последовательно выполняет три проверки. Если любая из них завершается ошибкой, выполнение прерывается, и клиенту возвращается статус `400 Bad Request` (отображаются нижние экраны ошибок на UI-схеме из ТЗ).

1. **Проверка обязательных полей:** Сервер проверяет, что внутри объекта `user` присутствуют ключи `first_name`, `last_name`, `email`, `password`, а их значения не равны `null` или `""`.
2. **Валидация формата Email:** Строка `user.email` проверяется на бэкенде с помощью регулярного выражения.
3. **Валидация сложности пароля:** Проверяется длина строки `user.password` (не менее 8 символов), наличие заглавной буквы и цифры.

## 2. Проверка уникальности учетных записей (Бизнес-логика)
Сервер исключает создание дубликатов аккаунтов. Выполняется поисковый запрос к СУБД:
```sql
SELECT id FROM users WHERE email = :email LIMIT 1;
```
* **Обработка результата:** Если СУБД возвращает запись, бэк-сервис останавливает регистрацию и возвращает клиенту ответ `409 Conflict` (экран ошибки «Пользователь уже существует»).

## 3. Криптографическая защита (Безопасность)
1. Сервер генерирует случайную криптографическую **соль (Salt)**.
2. На основе соли и чистого пароля вычисляется однонаправленный хэш с использованием алгоритма **BCrypt**. Хранение паролей в чистом текстовом виде в БД запрещено.

## 4. Транзакционная запись в БД и ответ системы
1. Бэк-сервис инициирует транзакцию в БД.
2. Выполняется SQL-запрос на вставку строки:
```sql
INSERT INTO users (first_name, last_name, email, password_hash, status, created_at)
VALUES (:first_name, :last_name, :email, :password_hash, 'active', CURRENT_TIMESTAMP);
```
* **Сценарий ошибки:** При техническом сбое транзакция откатывается (`Rollback`), клиенту возвращается статус `500 Internal Server Error`.
* **Успешный сценарий:** Транзакция фиксируется (`Commit`). Сервер собирает успешный JSON-ответ (подставляя сгенерированный `id`, `email` и статус `"active"`) и отправляет HTTP-ответ **`201 Created`** (на UI-схеме из ТЗ отображается финальный поп-ап успеха).

---

## 5. Структура таблицы Базы Данных (users)

| Имя поля | Тип данных | Ограничения (Constraints) | Описание |
| :--- | :--- | :--- | :--- |
| **id** | BIGINT / SERIAL | PRIMARY KEY, AUTO_INCREMENT | Уникальный идентификатор пользователя |
| **email** | VARCHAR(255) | UNIQUE, NOT NULL | Электронная почта (логин для входа) |
| **password_hash**| VARCHAR(255) | NOT NULL | Хэшированный пароль |
| **first_name** | VARCHAR(100) | NOT NULL | Имя пользователя |
| **last_name** | VARCHAR(100) | NOT NULL | Фамилия пользователя |
| **status** | VARCHAR(50) | NOT NULL, DEFAULT 'active' | Статус аккаунта |
| **created_at** | TIMESTAMP | NOT NULL, DEFAULT CURRENT_TIMESTAMP | Дата и время регистрации |
